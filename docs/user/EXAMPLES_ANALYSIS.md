# Examples analysis: runfabric-example vs scaffold

Analysis of [runfabric-example](https://github.com/runfabric/runfabric-example) and how to create examples for **all languages** in this repo.

## runfabric-example repo summary

- **Purpose:** Collection of boilerplates and serverless examples (AWS Lambda, Vercel, Netlify, Cloudflare, GCP, Azure, Alibaba, Fly, IBM).
- **Content:** `README.md` (example index by provider) and **`example.md`** (exact `runfabric init` command per example).
- **Important:** The repo lists **example folder names** and **init commands**; the actual project folders may be generated by running those commands (not necessarily committed).
- **Language in example.md:** All commands use **`--lang ts`** (TypeScript only). There are no Python, Go, Java, or .NET variants.

### Template usage in runfabric-example

| Template | Use case | Scaffold equivalent |
|----------|-----------|----------------------|
| `api`    | HTTP API  | `--template http` (alias) |
| `queue`  | Queue worker (SQS, etc.) | `--template queue` |
| `worker` | Storage / EventBridge / PubSub | Scaffold uses `--template storage`, `eventbridge`, or `pubsub` directly |
| `cron`   | Scheduled (cron) | `--template cron` |

## Scaffold init: supported languages and flags

From `engine/internal/cli/init.go`:

- **Languages:** `js`, `ts`, `python`, `go` (no Java or .NET in init).
- **Templates:** `http`, `cron`, `queue`, `storage`, `eventbridge`, `pubsub`. Aliases: `api` → `http`, `worker` → `queue`.
- **Flags:** `--dir`, `--template`, `--provider`, `--state-backend`, `--lang`, `--skip-install`, `--no-interactive`, `--with-build` (TS).

## Create-example commands for all languages

Use these from the **scaffold** CLI (`runfabric` from this repo). Each row is one example; repeat for each language you want.

### Base pattern

```bash
runfabric init --dir <DIR> --template <TEMPLATE> --provider <PROVIDER> --state-backend <STATE> --lang <LANG> --skip-install --no-interactive
```

Optional: add `--with-build` for TypeScript (adds `tsconfig.json` and `build` script).

### Language options

| Lang   | Use case |
|--------|----------|
| `js`   | Node.js (plain JavaScript) |
| `ts`   | TypeScript (add `--with-build` for tsc) |
| `python` | Python 3 |
| `go`   | Go |

### Example: AWS Lambda HTTP, all languages, state local

```bash
# JavaScript
runfabric init --dir runfabric-aws-lambda-http-state-local-js --template api --provider aws-lambda --state-backend local --lang js --skip-install --no-interactive

# TypeScript (with build)
runfabric init --dir runfabric-aws-lambda-http-state-local-ts --template api --provider aws-lambda --state-backend local --lang ts --with-build --skip-install --no-interactive

# Python
runfabric init --dir runfabric-aws-lambda-http-state-local-python --template api --provider aws-lambda --state-backend local --lang python --skip-install --no-interactive

# Go
runfabric init --dir runfabric-aws-lambda-http-state-local-go --template api --provider aws-lambda --state-backend local --lang go --skip-install --no-interactive
```

### Example: Cloudflare Workers HTTP + Cron, all languages

```bash
# HTTP
runfabric init --dir runfabric-cloudflare-http-state-local-ts --template api --provider cloudflare-workers --state-backend local --lang ts --skip-install --no-interactive
runfabric init --dir runfabric-cloudflare-http-state-local-js --template api --provider cloudflare-workers --state-backend local --lang js --skip-install --no-interactive

# Cron
runfabric init --dir runfabric-cloudflare-cron-state-local-ts --template cron --provider cloudflare-workers --state-backend local --lang ts --skip-install --no-interactive
```

### Example: GCP Functions (HTTP, Queue, Storage, PubSub)

```bash
# HTTP
runfabric init --dir runfabric-gcp-http-state-local-ts --template api --provider gcp-functions --state-backend local --lang ts --skip-install --no-interactive

# Queue
runfabric init --dir runfabric-gcp-queue-state-local-ts --template queue --provider gcp-functions --state-backend local --lang ts --skip-install --no-interactive

# Storage (use storage template; runfabric-example used worker then adapted)
runfabric init --dir runfabric-gcp-storage-state-local-ts --template storage --provider gcp-functions --state-backend local --lang ts --skip-install --no-interactive

# PubSub
runfabric init --dir runfabric-gcp-pubsub-state-local-ts --template pubsub --provider gcp-functions --state-backend local --lang ts --skip-install --no-interactive
```

### Naming convention (align with runfabric-example)

- **Pattern:** `runfabric-<provider>-<trigger>-state-<backend>[-<lang>]`
- **Examples:**  
  `runfabric-aws-lambda-http-state-local-ts`  
  `runfabric-aws-lambda-queue-state-local-python`  
  `runfabric-cloudflare-workers-http-state-local-js`

## Where to put generated examples in this repo

- **Node/TS/JS:** `examples/node/` (e.g. `examples/node/runfabric-aws-lambda-http-state-local-ts`).
- **Python:** `examples/python/`.
- **Go:** `examples/go/`.
- **Java / .NET:** When `runfabric init` supports them, add under `examples/java/` and `examples/dotnet/`.

You can add a script (e.g. `scripts/create-examples.sh`) that loops over provider × template × lang and runs `runfabric init` with the correct `--dir` under `examples/<lang>/`.

## Differences from runfabric-example

1. **Templates:** Scaffold has explicit `storage`, `eventbridge`, `pubsub`; runfabric-example uses `worker` then manual edit.
2. **Frameworks (Express/Nest/Fastify):** runfabric-example lists separate AWS examples for expressjs, nestjs, fastify. Those are **post-init** steps (add SDK and wire handler); init does not have a `--framework` flag.
3. **State backends:** Same options (local, s3, gcs, azblob, postgres); scaffold supports them via `--state-backend`.
4. **Languages:** runfabric-example only documents TS; this matrix adds js, python, go using the same init surface.
